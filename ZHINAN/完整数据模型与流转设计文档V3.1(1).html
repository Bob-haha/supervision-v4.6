📢 本次版本更新说明（V3.0 新增需求）
本版本在 V2.0 基础上，整合了以下新增需求和设计决策：
  • 首页态势多标签页与高级配置：关级、科级态势支持创建多个独立标签页，每个标签页可独立配置列选、筛选、排序。用户配置持久化存储，下次登录自动恢复。
  • 任务反馈负责人确认机制：子任务完成和环节推进产生的汇总信息，需经本级任务负责人确认后，才能作为本级任务的正式反馈内容对外展示。
  • 批量送阅：任务列表支持多选后一键批量送阅给指定人员。
  • P2P同步细粒度权限：任务仅创建人可删除。数据明细行仅创建人可删除和编辑本人创建的行，他人创建的行只读。
  • 长链条多部门协同解耦设计：通过"数据快照 + 引用关联 + 状态通知"机制，支持预算申报→采购申请→采购实施等跨部门长链条场景，各部门任务独立运作、数据解耦。
  • Word文档预览方案确定：采用 mammoth.js + pdf.js 纯前端方案，支持制度文件页码跳转。
新增 V3.0 增量更新（已整合至正文）： tasks 表新增 starredBy、activityLog 字段； metric_definitions.config 结构调整； user_dashboard_config 指标拆分； 流转模拟步骤增强。

本文档分为四大部分：

第一部分：数据模型定义

1.1 组织架构模块

1.1.1 部门表 departments

功能：存储关区所有科室、处室信息，构建组织树。支撑任务分派时部门选择、P2P定向同步时部门领导路由、态势按部门筛选。

使用页面：组织架构管理页、任务表单责任分配选择器、态势筛选。

责任人：小苏（管理页）、帅韬（选择器组件）、李彦鸿（P2P同步路由）。

字段名类型必填说明
idstring是UUID，部门唯一标识
namestring是部门全称（如"查验二科"）
shortNamestring否部门简称（如"查验二"）
parentIdstring否上级部门ID，为空表示一级部门，用于构建部门树
leaderIdstring否部门负责人ID，关联 personnel.id。任务分配到部门时，通过此字段找到默认接收人
ordernumber是排序号，控制同级部门显示顺序
isActiveboolean是是否启用，false时不在选择器和态势筛选中显示，但历史任务中的引用保留

关联与驱动：

  • leaderId → personnel.id：P2P同步时，系统查询任务主办部门对应的leaderId，将任务推送到该领导的设备。
  • parentId → departments.id：构建部门树，支撑态势按层级汇总。

1.1.2 人员表 personnel

功能：存储所有用户的基本信息、职务、所属部门。支撑任务分派时人员选择、P2P身份识别、权限判断。

使用页面：组织架构管理页、人员选择器、P2P连接身份映射、权限校验。

责任人：帅韬（管理页、选择器）、李彦鸿（与P2P peerId绑定）。

字段名类型必填说明
idstring是UUID，同时作为P2P网络中的peerId，客户端连接信令服务器时以此标识身份
namestring是姓名
employeeNostring是工号，唯一，用于Excel导入时的去重匹配
departmentIdstring是所属主部门ID，关联 departments.id。用于确定用户所属科室，支撑态势科室过滤
positionstring是职务（科长/副科长/科员/关长/副关长/督办员/财务人员/后勤人员等），用于前端展示和未来角色权限判断
isLeaderboolean是是否为其主部门的负责人。P2P同步时，任务分配到部门后，系统查询isLeader=true的人员作为默认接收人
emailstring否邮箱，用于浏览器扩展手动提取邮件时的发件人匹配
isActiveboolean是是否在职，离职人员不在选择器中显示
ordernumber是排序号

关联与驱动：

  • id 即 P2P peerId。用户打开系统时，浏览器扩展或Vue应用将此ID注册到信令服务器。
  • departmentId → departments.id：确定用户所属科室。
  • isLeader：决定是否为部门的默认任务接收人。

1.2 任务核心模块

1.2.1 任务表 tasks 🔴 更新

功能：存储每个工作任务的完整信息，是整个系统的核心数据载体。所有业务操作最终都体现为对此表某条记录的创建或更新。

使用页面：首页看板、个人工作台、任务列表、任务详情页、新建任务页、督办看板。

责任人：帅韬（详情页、新建页）、小苏（看板、列表）、李彦鸿（P2P同步、扩展联动）。

字段名类型必填说明
idstring是UUID，任务唯一标识
titlestring是任务标题，用于列表展示和搜索
descriptionstring否详细描述，富文本
taskTypeGroupIdstring否一级任务类型ID，关联 task_type_groups.id。用于分类筛选
taskTypeIdstring否二级任务类型ID，关联 task_types.id。创建时系统通过此字段自动挂载默认流程模板和明细模板
statusenum是任务状态：pending（待处理）、in_progress（进行中）、completed（已完成）、archived（已归档）
priorityenum是紧急程度：normal（普通）、urgent（紧急）、leadership_attention（领导关注）。标记为leadership_attention的任务在首页态势和关注列表中置顶
deadlinestring否截止日期，ISO 8601格式。用于临期/逾期判断（黄灯/红灯预警）
tagsstring[]否标签ID数组，关联 tags.id。支撑多标签筛选和督办分类汇总
parentTaskIdstring否父任务ID，为空表示顶级任务。建立父子关系时双向关联
childTaskIdsstring[]是子任务ID列表，由系统在创建子任务或关联已有任务时自动维护。用于快速查询任务树
sourceobject否任务来源。结构：{type: "manual"/"email"/"document", reference: "原文链接或文号"}。浏览器扩展提取时自动填充
responsibleMatrixobject是责任矩阵，记录主办部门/人员、协办部门/人员（结构见下方）
flowTemplateIdstring否创建时挂载的流程模板ID，关联 flow_templates.id
stagesSnapshotarray否环节快照数组，任务创建时从流程模板的stages拷贝而来。后续任务流转以此为准
currentStageIndexnumber否当前环节索引，指向stagesSnapshot数组中的某个元素。-1表示尚未开始或已结束
confirmedFeedbackobject否负责人已确认的反馈摘要。结构：{summary, confirmedAt, confirmedBy}
dataSheetTemplateIdstring否明细模板ID，关联 data_sheet_templates.id
dataSheetFieldsarray否明细字段定义快照。结构：[{fieldId, required, order}]
dataSheetRowsarray否明细数据行数组。每行结构：{rowId, data: {fieldId: value}, creatorId, createdAt, updatedAt}
childAggregationModeenum否父子任务数据汇总模式：none（独立）、homogeneous（同构）、heterogeneous（异构）
aggregationRulesarray否异构汇总规则，结构：[{sourceTaskId, sourceFieldId, targetSummaryId, aggFunction}]
snapshotsarray否关联上游任务的数据快照数组。用于长链条协同，冻结上游数据版本
extractionConfigobject否浏览器扩展提取配置（结构见下方）
authorizedPeersstring[]是有权访问此任务的peerId列表。P2P同步时仅向这些peer推送数据
sharedWithstring[]否送阅目标peerId数组。这些peer可查看任务详情但非直接责任人
createdBystring是创建人peerId。用于删除权限判断：仅createdBy可删除整个任务
createdAtstring是ISO 8601创建时间
updatedAtstring是ISO 8601最后更新时间，用于P2P增量同步判断
starredBy 🔴 新增 string[] 否 关注此任务的人员 peerId 数组。用于首页"关注任务"列表、任务详情页关注星标切换。默认空数组。
activityLog 🔴 新增 Array 否 任务动态记录数组，按时间顺序存储所有操作记录。每条记录为对象，结构见下方说明。默认空数组。

责任矩阵 responsibleMatrix 结构：

{
                "primary": [
                {"departmentId": "dept_01", "personnelId": "user_01", "role": "main"}
                ],
                "cooperative": [
                {"departmentId": "dept_02", "personnelId": null, "role": "assist"}
                ]
        }

personnelId为空时表示任务到达部门但尚未指派到具体人，部门领导需要进一步细化分派。当personnelId被填充时，该人员的所有设备收到任务同步。

环节快照 stagesSnapshot 结构：

[
                {
                "stageIndex": 0,
                "name": "制定方案",
                "options": [
                {
                "label": "提交科长审核",
                "nextStageIndex": 1,
                "resources": {
                "templates": ["file_id_001"],
                "links": ["http://..."],
                "policies": [
                {"fileId": "file_id_010", "clauseId": "clause_03",
                "clauseTitle": "第三章 查验规程", "pageNumber": 15}
                ]
                }
                }
                ],
                "requireAttachment": false,
                "requireSupervisorConfirm": false
                }
        ]

templates中的文件ID指向files表中的模板文件。policies中的文件ID指向files表中的制度文件，pageNumber用于pdf.js页码跳转。

提取配置 extractionConfig 结构：

{
                "enabled": true,
                "executorId": "user_05",
                "targetUrl": "https://lims.customs.gov.cn/result/123",
                "domMappings": [
                {"fieldName": "labResult", "selector": "#result-status", "attribute": "textContent"}
                ],
                "schedule": {"mode": "auto", "intervalMinutes": 60},
                "lastExtractionStatus": "success",
                "lastExtractionTime": "2026-05-28T08:00:00Z"
        }

🔴 activityLog 元素结构 新增：

{
                "id": "uuid",
                "type": "stage_progress | feedback | attachment | subtask_summary | leader_comment | system | responsibility_feedback | external_notification | urge",
                "content": "描述文本",
                "timestamp": "ISO8601 时间",

                "metadata": {

                "newStageIndex": 1,

驱动说明：

  • 环节推进、反馈提交、领导批示、催办、外部系统回写等操作，均通过向 activityLog 追加记录实现。前端动态区按此数组渲染时间线。
  • starredBy 更新时仅修改数组，不产生动态记录。

1.2.2 任务关联表 task_relations

功能：独立存储任务之间的关联关系及数据汇总模式。解除关联时仅删除此记录，不影响任务本身。支持父子任务和长链条引用两种关联类型。

使用页面：任务详情页子任务区域、数据明细汇总逻辑、长链条协同溯源。

责任人：李彦鸿（底层CRUD）、帅韬（详情页展示）。

字段名类型必填说明
idstring是UUID
parentTaskIdstring是父任务ID，关联 tasks.id
childTaskIdstring是子任务ID，关联 tasks.id
relationTypeenum是关联类型：parent_child（父子任务）、reference（引用关系，用于长链条协同）
aggregationModeenum是汇总模式：none、homogeneous、heterogeneous
aggregationRulesarray否异构汇总规则
createdAtstring是关联建立时间

🔗 关联时的权限合并：建立关联时（无论何种类型），系统自动将父任务的authorizedPeers与子任务的authorizedPeers双向合并，确保双方相关人员都能看到完整上下文。

1.3 流程与明细配置模块

1.3.1 流程模板表 flow_templates

功能：存储可复用的任务步骤序列及每个环节的操作配置。由管理员或科室负责人维护。

使用页面：流程配置页、新建任务页（选择流程）。

责任人：小苏。

字段名类型必填说明
idstring是UUID
namestring是模板名称（如"年度重点任务流程"）

createdBystring是创建人peerId

责任人：小苏。

字段名类型必填说明
idstring是UUID
namestring是字段显示名（如"预算金额"）
typeenum是字段类型：text（文本）、number（数字）、date（日期）、select（下拉选项）
optionsarray否当type为select时的选项列表
unitstring否单位（如"万元"、"台"）
isAggregatableboolean是是否可聚合。number和date类型默认为true
suggestedLevelenum否建议指标级别：customs_level（关级）、section_level（科级）
isSystemboolean是是否系统内置字段，内置字段不可删除

1.3.3 动态明细模板表 data_sheet_templates

1.5 文件管理模块

1.5.1 文件表 files

功能：集中存储所有模板文件、制度文件、任务附件。支持版本管理、条款目录、有效期、统一替换。制度文件存储条款目录用于pdf.js页码跳转。

使用页面：文件管理页、流程配置页、任务详情页、个人工作台。

责任人：帅韬（文件管理页、选择器）、李彦鸿（文件P2P同步逻辑）。

字段名类型必填说明
idstring是UUID，文件唯一标识
namestring是原始文件名
categoryenum是文件分类：template（模板）、policy（制度）、attachment（附件）
mimeTypestring是MIME类型
sizenumber是文件大小（字节）
dataBlob是文件二进制数据，直接存储在IndexedDB中
versionnumber是版本号，从1开始递增
isLatestboolean是是否最新版本
replacesFileIdstring否被替换的旧版本文件ID，形成版本链
relatedTaskIdstring否关联的任务ID（仅attachment时有效）
uploadedBystring是上传人peerId
uploadedAtstring是上传时间
制度文件特有字段（仅 category=policy 时有效）
policyNumberstring否文号（如"署监发〔2025〕12号"）
effectiveDatestring否生效日期
expiryDatestring否失效日期
clausesarray否条款目录数组，结构：[{id, title, pageNumber, children: [...]}]

📄 Word文档预览：采用mammoth.js将.docx转为HTML，再通过浏览器打印为PDF，使用pdf.js渲染。制度文件的clauses中pageNumber字段用于pdf.js的页码跳转。

📦 同步策略：template和policy类文件全关同步（P2P广播）；attachment类文件仅同步给关联任务的authorizedPeers。

                "mode": "filter" | "specific",

                // 当 mode = "filter" 时使用以下字段：
                "taskTypeIds": ["type_annual"],
                "filterTags": ["tag_leader"],
                "timeRange": {"start": "2026-01-01", "end": "2026-12-31"},

                // 当 mode = "specific" 时使用以下字段：
                "specificTaskIds": ["task_001", "task_002"],

                // 聚合定义（两种模式共用）：
                "fieldId": "sys_field_budget",
                "aggFunc": "SUM"
        }

说明：

  • mode 为必选项，确定使用"条件筛选"还是"指定具体任务"。
  • 条件筛选模式下，可组合 taskTypeIds、filterTags、timeRange，至少提供一个条件。
  • 具体任务模式下，specificTaskIds 至少包含一个任务 ID。
  • 聚合字段 fieldId 和函数 aggFunc 为两种模式共用，若 fieldId = "__task_count__" 则只能使用 COUNT。

第四部分：关键设计决策说明

决策点最终裁定理由
文件存储位置文件Blob直接存储在IndexedDB的files表中保证离线可用、安全保密、无需后端文件服务器
Word文档预览采用mammoth.js（转HTML）+ 浏览器打印为PDF + pdf.js渲染纯前端免费方案，支持页码跳转
制度文件条款跳转前端使用pdf.js渲染，传入pageNumber参数实现页码跳转稳定可靠，制度文件统一为PDF格式
环节资源引用锁定任务创建时快照文件ID，锁定版本。后续文件更新不影响已创建任务保证任务执行时使用的模板和制度版本一致，满足审计要求
父子任务权限合并建立关联时双向合并authorizedPeers；解除关联时不缩减确保关联双方能查看完整上下文，解除后保留已获得的查看权限
个人自定义指标存储在user_dashboard_config的customsMetrics/sectionMetrics数组中存储结构简洁，查询方便
督办员权限范围user_roles表中scope为空表示全关，否则限定指定部门灵活控制督办员的数据访问范围
送阅任务通过sharedWith数组存储被送阅人peerId无需额外建表，利用已有P2P同步机制
流程模板修改影响范围模板修改不影响已创建任务（任务存有环节快照）保证进行中的任务不受模板变更影响
长链条协同解耦采用"数据快照 + 引用关联 + 状态通知"模式各科室任务独立运作，通过快照冻结上游数据，避免下游阻塞上游，满足审计追溯
任务删除权限仅createdBy可删除整个任务防止误删，责任明确
数据明细行编辑权限仅creatorId可编辑和删除本人创建的行；可新增行保护数据完整性，防止越权修改他人填报的数据
负责人反馈确认子任务完成和环节推进产生待确认提示，负责人确认后更新confirmedFeedback确保正式反馈经责任人审核，避免信息噪音
态势多标签页在user_dashboard_config.tabs中存储，每个标签页独立保存列选、筛选、排序满足不同管理视角的灵活切换需求
P2P同步策略任务级定向同步，authorizedPeers控制可见范围；inspector角色可跨部门接收保证数据隐私，同时满足督办需求
🔴 新增 任务动态记录在tasks.activityLog数组中按时间顺序存储所有操作记录统一审计追溯入口，前端动态区统一渲染，避免分散存储
🔴 新增 态势指标筛选模式metric_definitions.config支持filter和specific两种模式兼顾灵活条件筛选和精准指定任务的场景需求
